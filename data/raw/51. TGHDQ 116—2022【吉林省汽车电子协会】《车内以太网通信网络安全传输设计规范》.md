# 引1 言

本文件要求主要依据GM/T0022《IPSeCVPN技术规范》、GM/T0024《SSLVPN技术规范》，按照我国相关信息安全政策和法规，结合实际应用需求和产品生产的实际经验，对车内以太网通信网络安全传输的设计开发、使用及管理提出针对性的规范要求。

车内以太网通信网络安全传输采用数字证书或密钥为应用系统提供身份验证、传输链路使用密钥加密，保证整个传输过程的真实性、机密性和完整性，本规范提出明确的技术要求，贴合安全传输的技术应用需求和管理需求，给予功能设计者指导方向。

网络数据包本身并不提供任何安全特性，很容易伪造出报文包的地址、修改其内容、重播以前的包以及在传输途中拦截并查看包的内容。安全传输协议提供了一种标准的、健壮的机制，可为P层及上层协议（如UDP和TCP）提供数据机密性、完整性、真实性等安全保证。

为了确保标准的规范化和普适性，本规范仅在身份认证、链路加密、数据防重放、证书密钥管理、密钥交换协商等安全技术要点进行规范要求，以形成通用性的设计规范。

# 5整体设计规范

# 5.1适用对象：本文件适用于车内系统的零部件之间使用以太网进行通信的场景，例如车内网关与智能驾驶系统之间的通信。

# 5.2整体设计规范：通信双方在通信前需要进行身份验证，确定身份后，通过建立的安全隧道进行数据报文通信。保证数据报文在网络上传输的机密性、完整性和防重放和不可否认性：a）安全传输总体要求包括通信交互流程的安全、消息防重放的安全和预置密钥数据的安全：b）客户端和服务端通信前进行身份认证，通过各自的密码数据（密钥、证书等）验证通信双方的身份。可以证实消息发送方是唯一可能的发送者，发送者不能否认发送过消息，防止其他人冒充：c）通信数据应采用会话密钥进行加密保护，会话通信密钥生命周期为当次会话有效，在通信时进行协商生成，会话结束密钥失效，下次通信重新协商。对机密数据进行加密保护，用密文的形式传输数据。第三方无法窃听；d）消息防重放，防止恶意用户通过重复发送捕获到的数据包所进行的攻击，即接收方会拒绝旧的或重复的数据包。通过通信计数器实现，每次通信报文发送后，通信计数器按照提前约定的增量递增。通信计数器当次会话有效，在通信建立时进行协商生成；e）密码数据为通信双方证明自身身份合法性、有效性的唯一数据，应确保该数据的写入安全和安全存储，确保不被篡改。

# 6功能设计规范

# 6.1身份验证

确保通信双方在身份真实可信的同时，提供的功能有数据源验证、数据完整性校验和防报文重放功能。实现车内数据的保密、完整、安全的传输。包括：

a）对于遵循IPSec协议的通信方，在算法协商阶段验证接入终端的证书及签名，并进行CRL的检查；b）对于遵循SSL协议的通信方，在每次SSL握手时，验证接入终端的证书及签名，并进行CRL的检查：c）支持单向、双向证书认证选择，设置接入终端是否需要提交证书；d）支持实时证书状态验证方式，如OCSP验证，或基于CA提供的其他接口进行实时证书状态验证；e）支持建立单向TLCP安全传输通道，并可收发数据；f）支持基于密钥及证书，建立双向TLCP安全传输通道，并可收发数据。

a）在基于证书的场景中，客户端首先将自身的证书发送给服务端；b）服务端接收到客户端的证书后，通过存储在安全芯片中的证书链验证证书的合法性，验证通过后将服务端证书发送给客户端：c）客户端接收到服务端的证书后，同样进行证书的验证；d）客户端验证完后，构建认证报文并发给服务端，认证报文使用服务端公钥加密，同时使用客户端的私钥进行签名：e）服务端接收到认证报文，使用客户端的公钥验签，使用服务端的私钥解密，获取客户端发送的数据；f）服务端构建认证报文并发送给客户端，认证报文使用客户端的公钥加密，使用服务端的私钥进行签名；g）客户端接收到认证报文，使用服务端的公钥验签，使用客户端的私钥解密，获取服务端发送的数据：h）双方使用密钥生成因子计算生成会话密钥，使用随机数计算生成计数器的初值：i）客户端和服务端的后续通信均用会话密钥进行保护，同时通信报文中增加计数器，每发送、接收一个报文，计数器按约定的增量递增：j）通信密钥、初始计数的生成算法应使用双因子密钥生成，通信双方均参与计算生成。

# 6.2链路加密：提供对业务报文的加解密功能，自动协商所使用的密码算法，以保障车内数据安全存储、车内通信安全，保障车内安全通信的机密性，以及保障控制器软件升级包的防泄露、防逆向破解等安全功能。数据加解密算法包括：a）支持对称算法的加密、解密；b）支持非对称算法的加密、解密。可依据项目实际需求灵活选择具体算法的类型。

# 6.2.1支持VPN

支持点到点之间建立VPN隧道，保证终端到终端之间的数据传输安全。

# 6.2.2消息防重放的安全设计：消息防重放的安全设计要求如下：a）防重放攻击采取计数器进行保护；b）计数器初值的计算依赖于通信双方交换的随机数。通信双方各自的随机数应由随机数生成器生成。交换随机数时通过对称密钥或证书的公钥进行加密，确保通信过程的保密性；c）双方通过密钥协商协议生成相同的计数器初值，计数器数值的生成算法与临时通信密钥的生成算法相同。每发送、接收一个报文，计数器数值按约定的增量递增。

# 6.3CRL管理

支持增加、查询、删除和更新CRL，更新CRL功能，支持以下三种模式：WEB站点自动下载模式、LDAP服务器自动下载模式和手工导入模式。

# 6.4随机数生成

随机数生成应采用硬件模块的随机数生成器，应符合GM/T0028的要求。

# 6.5密钥交换

具有密钥交换功能，通过协商产生会话密钥：

a）会话密钥生成过程通信双方均应参与，应使用随机数生成器随机生成密钥因子；进行加密保护：c）通信双方使用密钥生成因子，通过协商好的密钥生成算法生成相同的会话密钥。

# 6.6安全报文传输

安全报文传输要求如下：

a）对于遵循IPSec协议的安全传输应用设计，其报文封装格式应符合GM/T0022的要求：b）对于遵循SSL协议的安全传输应用设计，安全报文封装协议应符合GM/T0024的要求。

# 7预置密码数据安全

# 7.1密钥及证书存储安全

以下为密码数据存储安全要求细则：

a）密钥或证书对应私钥的存储位置视设备情况而定，有安全芯片的应存储在安全芯片中，没有安全芯片的应存储在TEE安全区域中；b）所有与密钥相关的计算都应在安全芯片中进行，保证密钥不出安全芯片或TEE安全区域；c）密钥或证书存储位置应保证只有安全组件能够进行写操作权限，应用程序只有读权限且操作需要进行验证，私钥不允许应用程序进行读取。

# 7.2密钥及证书安全写入

以下为密钥证书安全写入要求细则：

a）密钥或证书应与安全芯片序列号或零部件设备号一一对应，由车厂密码机中批量进行密钥证书数据的生成：b）密钥或证书在产线通过调用安全芯片厂商预留的接口注入到安全芯片中，注入过程应确保密钥或证书数据的保密性和完整性；c）对于安全芯片厂商不提供预留接口的场景，密钥或证书数据通过线下加密、签名的方式发送给安全芯片厂商，安全芯片提供者在安全芯片出厂时进行密钥或证书数据的写入，应确保写入过程的安全性；d）密钥或证书如果通过在线申请的方式从对应业务服务器中获取，获取通道应使用安全通信协议：例如安全强度大于或等于TLS1.2或GMSSLVPN的安全通信协议；e）对应业务服务器应从密码机中申请密码数据，密码机同时对密钥或证书数据进行签名和加密，确保完整性和保密性；f）接收到密钥或证书数据后应进行签名验证和解密，并存储到自身的安全区域，安全区域参考7.1节要求。

# 7.3密钥证书更新流程安全：以下为密钥证书数据更新流程安全要求细则：a）密钥证书数据如存储在安全芯片中，密钥证书数据更新应通过TSM对安全芯片进行更新时同步更新，或通过4S店换件进行更新；b）密钥证书数据更新，应先注销当前的密钥证书数据，再通过业务服务器进行密钥证书更新，过程与写入过程相同。

# 8性能要求规范

8.1认证及吞吐效率要求（主控制服务端）

每秒新建连接数不小于100次。要求最大并发连接数不小于600个。要求吞吐率不小于50Mbps。要求每秒认证数不小于100次。

# 8.2安全强度要求：对称密钥加解密安全强度不低于128位。非对称密钥加解密安全强度：采用RSASSA-PSS≥2048位，采用ECDSA≥256位。
