# 引1 開言

近年来，随着汽车与电子、软件、通信、人工智能、大数据等多个产业的跨界融合和协同创新，中国智能网联汽车已经逐渐形成了覆盖“车、路、云”的立体网状生态系统。这样车辆电气功能越来越复杂、和外界的通信越来越多，故车辆和外界通信方式越来越多样、车辆采用操作系统的控制器越来越多，故对车辆信息安全的开发及实现质量及水平要求越来越高。

远程通信模块作为车内外数据传输的中枢，其具备完善的linux或Android等操作系统，故暴露点多，故其最容易受到网络攻击。而远程诊断、远近程控车、蓝牙控车等多种功能均是以远程通信模块为主控节点进行功能实现的，只有远程通信模块实现了安全可靠的信息安全防护才能有效阻止黑客通过远程通信模块对车辆的攻击。

因此，有必要对远程通信模块的安全需求及测试方法进行系统规定，旨在提升远程通信模块的安全设计及实现水平，有效提升车辆的信息安全设计水平。

# 5.1.2.3设计要求

b）序，随后使用固件分析工具对其进行二进制逆向分析，判断反编译后的二进制代码是否混淆，如未做混淆处理则不满足5.1.2.3中b条款要求，反之满足5.1.2.3中b条款要求；

c）测试者使用固件分析工具对升级固件进行分析，如其中存在硬编码的密钥或控车、软件升级、功能认证等关键功能的算法，则不满足5.1.2.3中c条款要求，反之满足5.1.2.3中c条款

# 5.3 系统信息安全

5.3.1测试前提无。

# 5.3.3设计要求

智能网联汽车远程通信模块系统信息安全设计要求如下：

a） 系统登录时需要进行身份校验；b） 系统登录时不能使用常见的弱口令密码，且密码不能与账号相同：c）系统身份验证口令应至少包含数字、大写字母、小写字母、特殊字符中的两种以上且密码长度大于8位；d）系统不应存在由权威漏洞平台公开发布时间在六个月以上的高危安全漏洞；(e 系统应关闭非设计文档要求的业务端口；f）系统应该存在防root机制，并在系统被非法root之后，给出安全警报，且无提权漏洞；g） 系统uboot应该具有防篡改机制，系统应存在uboot校验机制；h） 系统内核应该具有防篡改机制，系统应存在内核校验机制；i） 系统应具备权限访问控制功能，禁止安装外来非法应用；j）k）系统应存在ASLR缓冲区溢出安全保护技术，且对应的保护技术应处于开启状态。

# 5.3.4测试方法及判断依据

智能网联汽车远程通信模块系统信息安全测试方法及要求如下：

a）测试者使用连接工具通过各调试口访问被测件命令行，核查登录时有无身份校验。若有则满足5.2.3中a条款要求，反之不满足5.2.3中a条款要求；b）测试者使用连接工具通过各调试口访问被测件命令行，尝试使用常见弱口令字典对密码进行爆破，若能通过无线枚举的方法实现爆破且密码与账号不同则不满足5.2.3中b条款要求，反之满足5.2.3中b条款要求；c）测试者使用连接工具通过各调试口访问样件命令行：核查登录样件命令行身份校验的口令策略，如包含数字、大写字母、小写字母、特殊字符中的两种及以上且密码长度大于8位，则满足5.2.3中c条款要求，反之不满足5.2.3中c条款要求；或通过外置有线网卡接入被测件U口的方式连接被测件，在漏洞扫描工具上创建漏洞任务对被测件进行扫描并记录扫描结果。若不存在权威漏洞平台公开发布的六个月以上高危漏洞，则满足5.2.3中d条款要求，反之不满足5.2.3中d条款要求；e）将配置有端口扫描工具的测试设备通过被测件WIFI的AP或STA模式连接被测件，或通过外置有线网卡接入被测件USB接口的方式连接被测件，测试者通过端口扫描工具对被测件系统进行端口扫描，若发现非设计文档要求的业务端口开启则不满足5.2.3中e条款要求，反之满足5.2.3中e条款要求；f）测试者通过USB调试模式（常见的为ADB模式）将配置有Root检测工具的测试设备与被测件连接并对其进行root操作，如在确认root成功后而系统确无安全报警的则不满足5.2.3中f条款要求，反之满足5.2.3中f条款要求：g）测试者对固件中的uboot部分代码进行篡改并将篡改后的代码刷入被测件，在完成刷写后尝试上电重启，如可以正常重启则不满足5.2.3中g条款要求，反之满足5.2.3中g条款要求；h）测试者对固件中的内核部分代码进行篡改并将篡改后的代码刷入被测件，在完成刷写后尝试上电重启，如可以正常重启则不满足5.2.3中h条款要求，反之满足5.2.3中h条款要求；i）测试者使用连接工具通过各调试口访问被测件命令行；检测是否可按所有者、组和其他用户来设置读、写和执行保护；检测所有者是否可为特定用户或组单独指定权限检查并设置允许或禁止安装外来非法应用。若未区分不同用户间访问控制权限或被测件可被直接安装非法应用成功，则不满足5.2.3的i条款，反之满足5.2.3的i条款；j）检测系统防火墙设置流量访问控制策略，如果采用缺省的全禁止或全允许规则不满足5.2.3的条款，反之满足5.2.3的j条款；k）测试者使用连接工具通过各调试口访问被测件命令行：检测系统是否具有ASLR缓冲区溢出安全保护机制，如有则满足5.2.3的k条款要求，反之不满足5.2.3的k条款要求。

# 5.4通信连接信息安全

# 5.4.1测试前提

在进行智能网联汽车远程通信连接信息安全测试前，应准备如下信息：

a）测试件总线方式对其对应的引脚定义；b）调试接口的接口定义文档。

# 5.4.2测试环境

测试环境参见本规范5.2.2，使用其中的通信协议分析工具。

# 5.4.3设计要求

智能网联汽车远程通信模块通信连接信息安全设计要求如下：

a）与外部通信需采用TLS1.2及以上协议；b）不应开放危险访问端口，若由于功能需要确需开放的必须设计安全访问控制机制；c）不能存在可以监听反弹shell的后门程序；d）远程通信模块应和车内其他零部件进行隔离，且禁止从其他零部件跳转调试接口进入远程通信模块：e）应具备安全可靠的27访问机制且其随机seed库重复率要低于1/10000；f）当车速小于3Km/h时不允许对车辆进行10复位操作；g）通信及诊断中的关键信息需采用密文传输或存储；h）重要通信数据需进行完整性校验；i）重要的自动驾驶、底盘控制及动力控制等通信数据需进行来源真实性校验。

# 5.4.4测试方法及判断依据

智能网联汽车远程通信模块通信连接信息安全测试方法及要求如下：

a）测试者使用抓包工具对远程通信模块和云端的通讯数据进行抓包并分析数据，对采用TLS1.2及以上协议进行了加密则满足5.3.3中a条款要求，反之不满足5.3.3中a条款要求；b）将配置有端口扫描工具的测试设备通过被测件WIFI的AP或STA模式连接被测件，或通过外置有线网卡接入被测件U口的方式连接被测件，测试者通过端口扫描工具对被测件系统进行端口扫描，若发现22、23、80等端口且该端口未作访问控制认证机制，则不满足5.3.3中b条款要求，反之满足5.3.3中b条款要求；c）测试者进入系统shell并查看nc程序，如存在nc－e参数后门则不满足5.3.3中c条款要求；查看系统进程及后台是否有应用监听，判断是否存在挂马文件，如存在则不满足5.3.3中c条款要求：反之满足5.3.3中c条款要求；d）测试者通过测试件shell查询车内其他Eth通信部件IP地址，并通过远程通信模块和车内除网关外其他以太网节点进行通信连接，如远程通信模块能PING通车内除网关外其他以太网节点，或通过除网关外其他以太网节点能直接访问远程通信模块，则不满足5.3.3中d条款要求，反之满足5.3.3中d条款要求；e）测试者对被测件进行大量27访问，从其回复的SEED中分析其随机性程度，如重复率低于1/10000则满足5.3.3中e条款要求，反之不满足5.3.3中e条款要求；f）测试者在车速小于3km/h且通过27安全访问的情况下对被测件进行11服务操作，如复位成功则不满足5.3.3中f条款要求，反之满足5.3.3中f条款要求；g）测试者对被测件传输报文及存储固件进行监控，判断其中的密钥、安全认证随机数等重要的数据是否采用密文存储和传输，如是则满足5.3.3中g条款要求，反之不满足5.3.3中g条款要求：h）测试者对被测件接收的和驾驶姿态强相关的报文进行篡改，如被测件能够发现该报文被篡改过而丢弃则满足5.3.3中h条款要求，反之不满足5.3.3中h条款要求；i）测试者对原本由节点A发送给被测件的重要自动驾驶、底盘控制及动力控制等通信数据进行截取，并通过伪造的节点B发送给被测件，如被测件能发现并将报文丢弃则满足5.3.3中i条款要求，反之不满足5.3.3中i条款要求。

# 5.5.1.3设计要求

智能网联汽车远程通信通用数据信息安全设计要求如下。
