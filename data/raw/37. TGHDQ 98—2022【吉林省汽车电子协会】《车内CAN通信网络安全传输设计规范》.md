# 引1

近年来，汽车电子的发展突飞猛进，车辆搭载的电子控制单元数量迅速增长，规模逐渐庞大。CAN总线作为车内网络中最主要的通信协议，其设计之初的核心理念是低成本和高稳定性。但CAN本身是没有考虑到信息安全问题的，如明文传输、广播传输、无内容校验等特性，这使得车内网络轻松便可受到攻击。

因此，有必要编制一个基础性标准为传统CAN总线引入一套安全传输的方法，旨在对车内CAN通信网络安全传输进行规范化要求，统一设计要点，形成平台化技术要求，以保证其设计的安全性和过程一致性。

为了确保标准的规范化和普适性，本规范主要对交互过程、密钥管理、MAC值构造、新鲜度值构造及同步等安全技术要点进行要求和规范，以形成通用性的技术要求。

# 车内CAN通信网络安全传输设计规范

# 1范围：本文件规定了车内CAN通信网络安全传输的设计规范，包括CAN通信网络安全传输交互过程、密钥管理、MAC值构造、新鲜度值构造及同步等安全要求。本文件适用于车内CAN总线协议的安全通信业务，旨在指导此类业务的信息安全技术设计、实现、验证、生产等工作。

# 5CAN通信网络安全传输总体要求：基于密码学技术，SecOC作为车内CAN通信网络传输的主要安全机制，能够为车内CAN通信网络传输的完整性（含真实性、新鲜度）提供保障。为了实现上述目标，MAC被附加到总线消息上。这些MAC为消息的完整性提供保护：a）应识别与信息安全相关的报文；b）每个发送方ECU应通过应用密码学技术，来保护与信息安全相关报文或信号的完整性，含真实性和新鲜度：c）每个接收方ECU应通过应用密码学技术，来验证收到的与信息安全相关报文或信号的完整性，含真实性和新鲜度。

# 6CAN通信网络安全传输流程安全要求

# 6.1概述：对CAN通信网络中传输的需进行通信安全加固的报文，发送方使用车内安全通信密钥对报文数据和验证该MAC值。当且仅当产生的MAC值相等时，验证才会成功。

# 6.3发送端安全传输技术流程：以下为车内CAN通信发送端安全传输流程及SecOC模块工作流程，如图3所示：a）发送端标识要传输的安全报文；b）发送端FVM中得到FV，将FV附加到PDU中；c）发送端基于FV和需要保护的PDU，使用车内安全通信密钥计算MAC；d）发送端传输真实和受重放保护的报文。

# 6.4接收端安全传输技术流程：以下为车内CAN通信接收端安全传输流程及SecOC模块工作流程，如图4所示。a）接收端将传入报文标识为安全报文并解析；b）FVM判断FV是否完整，若完整，将接收到的FV与本地的作比较，得到验证用新鲜度值；若不完整，比较接收到的FV低位与上一验证成功的FV低位的大小，并构造本次验证用新鲜度值；c） 接收端基于FV和需要保护的PDU，使用车内安全通信密钥计算MAC：d）比对计算得出的MAC与接收到的MAC值是否一致，验证通过则安全报文通过验证，验证不通过则丢弃该报文并记录日志。

# 7密钥管理安全要求：密钥管理系统持有车内安全通信密钥，为实现跨ECU车内安全通信，应对每个ECU安全的注入车内安全通信密钥，这涉及到与密钥管理系统互连。同时，应便于维修人员更换ECU时，将密钥进行更新，使一组车内安全通信ECU始终共享有效的车内安全通信密钥。以下为密钥管理安全要求细则：a）应管理和维护密钥管理系统，系统应具备注入和更新车内安全通信密钥，以及读取和处理故障日志的功能；b) 应确保密钥管理系统安全存储车内安全通信密钥及故障日志；c） 初始的车内安全通信密钥应安全的注入并存储至车内安全通信ECU，应保证生产环境安全可信：d）应确保ECL和密钥管理系统之间的可信连接。

# 8MAC值构造安全要求

MAC算法采用对称加密方式，应使车内CAN通信发送端与接收端具备相同的密钥，安全算法实现可采用纯软方式实现或内置HSM硬件实现。

以下为MAC认证码安全要求细则：

a）MAC应根据报文标识符、报文数据和FV生成；

b） 密钥长度应不低于128位，如使用AES-128-CMAC算法；

c）有限有效载荷的通信协议，如CAN总线，根据性能要求，建议采用截断的MAC。

# 9新鲜度值构造及同步安全要求

# 9.1总体要求

每个安全报文应包含用于确保安全报文新鲜度的单调计数器。这种单调计数器建议通过多计数器的方式实现。新鲜度值来自于FVM，应安全存放。

以下为新鲜度值安全要求细则：

a）不同的安全报文，宜配置单独的FV；b）应引入一项功能，以保证参与SecOC的每个ECU同步FV；c）应选取发送端ECU，其应能到达所有参与SecOC的ECU，用于FV消息同步；d）FV同步消息应周期性发送给所有参与SecOC的ECU。

# 9.3计数器同步设计要求

# 9.3.1总则：发送端、接收端ECU的FVM应具有新鲜度值同步功能，用于将发送端ECU的FVM的当前新鲜度值信息同步到接收端ECU的FVM中，保证安全报文收发两端新鲜度值的一致性，

# 9.3.2同步报文结构定义

新鲜度值同步报文结构定义如图6所示。

以下为新鲜度值同步报文结构定义安全要求细则：

a）同步计数器和重置计数器应为发送端ECU的FVM的当前值；b）接收端ECU的FVM应通过MAC值信息验证新鲜度值同步报文的真实性。

# 9.3.3同步请求报文结构定义

新鲜度值同步请求报文应包含同步请求标识。

# 9.3.4同步请求报文发送时机：以下为新鲜度值同步请求报文发送时机安全要求细则：a）接收端ECU应在完成初始化，但一定时间内未收到同步报文时发送；b）接收端ECU应在发生BUSOFF，并检测到BUSOFF恢复时发送。

# 9.3.5同步报文发送时机

以下为新鲜度值同步报文发送时机安全要求细则：

a）发送端ECU应在完成初始化，并在网络管理报文发出后发送；b） 发送端ECU应根据重置周期的设置周期性的发送；C） 发送端ECU应在收到接收端ECU发送的新鲜度值同步请求报文时发送；d） 发送端ECU应在发生BUSOFF，并检测到BUSOFF恢复时发送。

# 9.3.6同步报文处理技术流程

以下为新鲜度值同步报文处理流程安全要求细则：

a) 发送端ECU获取当前的同步计数器和重置计算器；

b）使用车内安全通信密钥计算MAC值信息；

c）按照新鲜度值同步报文结构构建同步报文并发送；

d）接收端ECU接收同步报文后：

1）使用车内安全通信密钥校验MAC值信息，若校验失败则丢弃该报文并记录日志；2）比较接收到的与本地保存的同步计数器与重置计数器；3）验证通过后，将接收到的同步计数器与重置计数器同步到本地，若验证失败则丢弃该报文并记录日志。
