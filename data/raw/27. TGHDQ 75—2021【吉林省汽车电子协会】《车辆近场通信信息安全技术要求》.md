近年来，我国智能网联汽车产业高速发展，便捷化、实用化功能越来越受客户欢迎。智能网联汽车越来越多的业务场景采用近场通信的方式实现，比如数字钥匙、无感锁车、近场控车等。在新的近场通信业务上车前，应进行相应的信息安全技术设计。这样遇到一个业务场景进行一个场景的安全设计，不但不成系统，也容易造成信息安全设计点的遗漏。

因此，有必要编制一个基础性标准对近场通信业务的信息安全设计进行规范化要求，统一设计要点，形成平台化技术要求，以保证近场通信业务信息安全设计的安全性和过程一致性。

为了确保标准的规范化和普适性，本规范仅在身份认证、通信保护、数据防重放、初始密码数据写入等安全技术要点进行要求和规范。以形成通用性的技术要求。

# 车辆近场通信信息安全技术要求

# 1范围

本文件规定了车辆近场通信信息安全的技术要求，包括近场通信安全的交互流程、密钥协商要求、防重放要求等，车辆近场通信包括但不限于蓝牙、WI-FI、NFR等。本文件仅针对承载在车辆近场通信协议上的业务部分进行安全设计，不对蓝牙、WI-FI、NFR等近场通信协议本身进行要求。

本文件适用于承载在车辆近场通信协议上的业务，指导此类业务的信息安全技术的设计开发、验证、生产等工作。

# 3术语和定义

3.1重放攻击Replay Attacks：重放攻击是指攻击者通过记录通信会话，并在以后某个时刻重放整个会话或者会话的一部分，以达到欺骗系统的目的。

3.2密钥因子Cryptographic Factor:密钥因子是指通过特定的数学计算，用来生成密钥的因子。

3.3数字证书Digital Certificate:也称公钥证书，简称证书，有证书认证机构（RA）签名的包含公开密钥拥有者信息、公开密钥、签发者信息、有效期以及扩展信息的一种数据结构。按类别可分为个人证书、机构证书和设备证书，按用途可分为签名证书和加密证书。

[来源：GM/T0024-2014SSLVPN技术规范，3.1]

3.4安全芯片Secure Element:含有密码算法、安全功能，可实现密钥管理机制的集成电路芯片。

# 5近场通信信息安全总体要求

以下为近场通信信息安全总体要求的细则。

a）近场通信信息安全总体要求包括近场通信交互流程的安全、消息防重放的安全和预置密码数据的安全。

b）近场通信涉及到移动端（手机、手环、pad等智能设备）及车端，双方应在通信前进行身份认证，通过各自的密码数据（密钥、证书等）验证通信双方的身份。

c）通信数据应采用临时通信密钥进行加密保护，临时通信密钥生命周期为当次会话有效，在通信时进行协商生成，会话结束密钥失效，下次通信重新协商。

d）消息防重放通过通信计数器实现，每次通信报文发送后，通信计数器按照提前约定的增量递增。通信计数器当次会话有效，在通信建立时进行协商生成。

e）密码数据为通信双方证明自身身份合法性、有效性的唯一数据，应确保该数据的写入安全和安全存储，确保不被篡改。

# 6.2基于证书的近场通信安全认证流程

以下为基于证书的近场通信安全认证流程：a）在基于证书的场景中，移动端首先将自身的证书C1发送给车端。b）车端接收到移动端的证书C1后，通过存储在安全芯片中的证书链验证证书的合法性，验证通过后将车端证书C2发送给移动端。c）移动端接收到车端的证书C2后，同样进行证书的验证，如图2所示。d）移动端验证完C2后，构建认证报文1（包含密钥生成因子K1，随机数R1)，并发给车端，认证报文1使用车端公钥加密，同时使用移动端的私钥进行签名。

# 6.3临时通信密钥生成安全要求

以下为临时通信密钥生成安全要求细则：a）临时通信密钥生成过程应移动端、车端双方均参与，应使用随机数生成器随机生成密钥因子K1/K2。b）密钥生成因子应进行保密性保护，对称密钥场景使用对称密钥加密：证书场景使用对方的公钥进行加密保护。c）通信双方使用密钥生成因子，通过协商好的密钥生成算法生成相同的临时通信密钥K，如：DH算法等。

# 7消息防重放安全要求

以下为消息防重放安全要求细则：a）本文中防重放攻击采取计数器R进行保护。b）计数器R的计算依赖于移动端和车端交换的随机数R1、R2。交换过程通过对称密钥或证书的公钥进行加密，确保通信过程的保密性，计数器R的生成算法与临时通信密钥K的生成算法相同。c）移动端、车端应有随机数生成器在交互过程中生成，双方通过密钥协商协议生成相同的计数器R，每发送、接收一个报文，计数器R按约定的增量递增。

# 8预置密码数据安全

# 8.1密码数据存储安全要求

以下为密码数据存储安全要求细则：a）车端密码数据（密钥、证书）一车一密，密钥或证书应存储在近场通信控制器的安全芯片中，所有与密钥相关的计算都应在安全芯片中进行，保证密钥不出安全芯片。b）移动端密码数据（密钥、证书）一机一密，存储位置视移动端设备情况而定，有安全芯片的必应存在安全芯片中，没有安全芯片的应存储在TEE安全区域或白盒中。c）密码数据存储位置应保证只有安全组件能够进行写操作权限，应用程序只有读权限且操作需要进行验证；私钥不允许应用程序进行读取。

# 8.2车端密码数据安全写入要求

以下为车端密码数据安全写入要求细则:a）车端密码数据应与安全芯片的序列号一一对应，由车厂加密机中批量进行车端密码数据的生成。b）保密码数据的保密性和完整性。c）对于安全芯片厂商不提供预留接口的场景，车端密码数据通过线下加密、签名的方式发送给安全芯片厂商，安全芯片提供者在安全芯片出厂时进行车端密码数据的写入，应确保写入过程的安全

# 8.3移动端密码数据安全写入要求

以下为移动端密码数据安全写入要求细则：a）移动端的密码数据应通过在线中请的方式从对应业务服务器中获取，获取通道应使用安全通信协议，例如安全强度大于或等于TLS1.2或GMSSLVPN的安全通信通信协议。b）对应业务服务器应从加密机中中请移动端的密码数据，加密机同时对密码数据进行签名和加密确保完整性和保密性。c）移动端接收到密码数据后应进行签名验证和解密，并存储到自身的安全区域。

# 8.4密码数据更新流程安全要求

以下为密码数据更新流程安全要求细则:a）车端密码数据均存储在安全芯片中，密码数据更新应通过TSM对安全芯片进行更新时同步更新，或通过4S店换件进行更新。b）移动端密码数据更新，应先注销当前的密码数据，再通过业务服务器进行密码数据更新，过程与写入过程相同。
